FINDING E-1 | SEVERITY: P1 | FILE: socio_sim/content/llm_cache.py:98
ISSUE: Tamper detection only runs when `record_hash` is present. A dict carrying `status:"accepted"` (or a forged blocked→accepted) with the `record_hash` key *deleted* skips the integrity check entirely and is served verbatim. This re-opens the exact P0 the module was built to close (R-CLAUDE-P0): flip a blocked entry to `accepted` **and delete `record_hash`**, and the blocked LLM text is served as accepted. Every `make_entry` writes a `record_hash`, so a status-bearing dict lacking one is never legitimate.
EVIDENCE:
```python
if "record_hash" in cached:            # skipped when the key is absent
    expected = record_hash(text or "", status or "", reasons)
    if cached["record_hash"] != expected:
        return CacheLookup(hit=False, ... "cache_tampered:record_hash_mismatch")
...
# status == "accepted".
return CacheLookup(hit=True, text=text, degradation=None)
```
FIX: before the `if "record_hash" in cached:` block →
```python
if status in _KNOWN_STATUSES and "record_hash" not in cached:
    return CacheLookup(hit=False, text=None,
        degradation="cache_tampered:missing_record_hash; entry discarded and regenerated")
```
(legacy status-less dicts still fall through to the existing re-screen path).
TEST: extend `test_tampered_cache_entry_is_discarded_and_not_served` — after forging `status="accepted"`, also `del cache[key]["record_hash"]`; assert transport is re-called and "email me" never appears.
